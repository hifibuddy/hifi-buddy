# M1.1 — Server-Side Config (Implementation Notes)

Status: **server-side + frontend complete, awaiting in-browser verification.**

## What changed

### `server.py`
- New module-level constants: `HIFIBUDDY_HOME` (defaults to `~/.hifi-buddy/`, override via `HIFIBUDDY_HOME` env var), `CONFIG_PATH`, `CONFIG_ALLOWED_KEYS` whitelist.
- New helpers: `load_config()`, `save_config_atomic()` — atomic `.tmp` + `os.replace`, dir chmod 700, file chmod 600.
- New endpoints:
  - `GET /api/config` — returns the full config object (whitelist-filtered).
  - `POST /api/config` — partial-update merge. Send `{key: value}` to set, `{key: ""}` or `{key: null}` to remove. Returns the new full config.
  - `POST /api/config/reveal` — opens `~/.hifi-buddy/` in Finder/Explorer/xdg-open. Returns `{opened, path}`.
- Startup banner now prints the config path so users always know where their data lives.
- Top-of-file docstring updated.

### `js/settings.js`
- New `CONFIG_KEY_MAP`: localStorage key → server config key. Tier A keys go through this; everything else stays in localStorage.
- New module-level state: `_configCache`, `_configReady`, `_writeQueue` (serializes server writes).
- `loadConfigFromServer()` — fetched once at boot; populates `_configCache`.
- `migrateLocalStorageToServer()` — one-shot, runs after config load. Reads any leftover Tier A keys from localStorage, POSTs them to the server, removes them from localStorage. Idempotent (subsequent boots are no-ops).
- `get()` / `set()` / `remove()` — transparently route Tier A keys to the cache + queued server writes. Non-Tier-A keys still hit localStorage as before. **Critical fallback**: `get()` falls back to localStorage when the cache is empty, so settings never appear "lost" if the server is unreachable or running older code without the new endpoints.
- `init()` is now `async` — runs the legacy `musictrip_*` migration, awaits `loadConfigFromServer()`, then awaits `migrateLocalStorageToServer()`, then binds the settings gear.
- New public exports: `getOllamaUrl`, `getOllamaModel`, `setSpotifyAuthMethod`, `revealConfigFolder`.
- Backup & Restore section in the settings modal replaced with a single "Reveal config folder" button. The export/import handlers are gone — backup is now `cp -R ~/.hifi-buddy/ <somewhere>`.
- KEYS constant gained `ollamaUrl` and `ollamaModel` entries (these were previously accessed via raw `localStorage.getItem('hifibuddy_ollama_*')` — that bypass is now closed).

### `js/app.js`
- `initializeModules()` and `init()` are now `async`. Settings initialization is awaited before any other module's init runs, since other modules call `HiFiBuddySettings.getXxx()` synchronously during their own boot.

### `js/lesson-generator.js`
- `callAI()` reads Ollama URL/model via `HiFiBuddySettings.getOllamaUrl/Model()` instead of raw localStorage.

### `js/hifi-buddy.js`
- `callAI()` likewise.
- One `localStorage.setItem('hifibuddy_spotify_auth_method', 'pkce')` rewritten to `HiFiBuddySettings.setSpotifyAuthMethod('pkce')`.

### `index.html`
- Cache-bust versions: `settings.js?v=16`, `app.js?v=11`, `lesson-generator.js?v=11`, `hifi-buddy.js?v=16`.

- User-created lessons

## End-to-end verification (left for the user)

1. **Restart the server** so it picks up the new server.py code:
   ```
   # In whichever terminal has python3 server.py running, Ctrl-C
   cd ~/code/hifi-buddy/hifi-buddy-app
   python3 server.py
   ```
   Startup should now print:
   ```
   HiFi Buddy server running at http://127.0.0.1:8090/
   Config:  /Users/laramies/.hifi-buddy/config.json
   ```

2. **Reload the app** with hard refresh (Cmd-Shift-R). Console should log:
   ```
   [HiFi Buddy] Migrated N keys from localStorage to ~/.hifi-buddy/config.json
   ```
   on first reload after the upgrade. N should be ~5–9 depending on what you have configured.

3. **Verify the config file**:
   ```
   cat ~/.hifi-buddy/config.json
   ls -la ~/.hifi-buddy/
   ```
   File should be `chmod 0600`, dir should be `chmod 0700`.

4. **The big test — clear site data, settings survive**:
   - DevTools → Application → Storage → "Clear site data" (check all)
   - Reload the page (Cmd-Shift-R)
   - Open Settings — Plex URL/token, Spotify Client ID, Claude key, etc. should all still be filled in.
   - Test Plex Connection — should still work.
   - Spotify connection → there's a quirk: the OAuth tokens themselves are Tier D and DO get cleared. You'll need to click "Connect Spotify" to re-auth. But the Client ID is preserved so you don't have to re-paste it.

5. **"Reveal config folder" button**: Settings → "Config & Backup" section → click "Reveal config folder". Finder should open showing `~/.hifi-buddy/`.

6. **Defensive case — server down**: Stop the server, reload the app. Settings should still be readable (falls back to in-memory cache + localStorage during the brief mismatch window). New writes won't persist until server is back. Console will log a warning.

## Known limitations / follow-ups

- The Spotify OAuth tokens stay in localStorage (they're short-lived; clearing site data forces a re-auth which is fine).
- ABX results and timing-feedback corrections are still in localStorage — they move in **M1.2** (next milestone).
- The Plex match cache (~500 KB) stays in localStorage — by design (per Sprint 1 plan; cache migration was deferred indefinitely as Tier C).
- If the user is running multiple browser windows simultaneously and changes the same key in both, last-writer-wins per key (server merges per-key on POST, so other untouched keys aren't overwritten).
- Migration is one-way: localStorage → server. There's no "go back" mode. If the user wants to revert to the localStorage-only build, they'd need to git revert + manually copy values back. Acceptable for this version.
